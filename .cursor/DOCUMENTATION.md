# Документация проекта

Данный проект должен представлять собой клиент-серверное веб-приложение. Поскольку данный проект создаётся для научной работы, его дальнейшее описание будет отчасти академическим, как описание научной работы. Ниже будет представлено его более подробное описание, включая тематику и требования.
В условиях активного развития цифровых образовательных технологий и широкого использования онлайн-обучения возрастает потребность в создании образовательных информационных систем, способных адаптироваться к индивидуальным особенностям учащихся. Современные образовательные платформы предоставляют значительное количество учебных материалов, но их выбор и последовательность изучения часто определяются заранее или на основе упрощённых механизмов оценки знаний. Такой подход не всегда позволяет учитывать реальные трудности учащихся и динамику обучения, что снижает эффективность образовательного процесса и мотивацию пользователей.
Тестирование остаётся одним из ключевых инструментов оценки уровня знаний в образовательных системах. В большинстве существующих решений результаты тестирования интерпретируются как агрегированные показатели, такие как общий балл. В то же время подробная информация о процессе выполнения задания, включая характеристики отдельных ответов, время реакции и иные показатели обычно не затрагиваются. Это приводит к недостаточно точному определению уровня владения отдельными темами и, как следствие, к формированию рекомендаций по учебным материалам, которые не в полной мере удовлетворяют потребности конкретного пользователя.
Научные исследования представляют различные подходы к адаптивному обучению и рекомендательным системам, включая методы машинного обучения, интеллектуальные обучающие системы и алгоритмы совместной фильтрации. Несмотря на свою эффективность, многие из этих подходов требуют значительных вычислительных ресурсов и объёмов обучающих данных. Это затрудняет их использование в системах с ограниченным числом пользователей или на ранних стадиях работы системы. В связи с этим важно разработать алгоритм, повышающий качество рекомендаций, основанных на анализе результатов тестирования, без необходимости использования сложных моделей обучения.
Целью данного проекта является разработка клиент-серверного веб-приложения, в которое будет интегрирован разработанный алгоритм интеллектуального подбора учебных материалов для обучающихся. Данное приложение предоставит возможность для преподавателя добавлять новые тесты и материалы, а также возможность для обучающегося проходить тесты и получать рекомендации для изучения материала.
К разрабатываемому приложению предъявляются следующие функциональные требования:
•	приложение должно давать возможность преподавателю редактировать учебные темы;
•	приложение должно давать возможность преподавателю редактировать учебные материалы;
•	приложение должно давать возможность преподавателю редактировать состав тестов и их вопросов;
•	приложение должно давать возможность преподавателю детально настраивать тесты и вопросы, включая сложность вопроса и нормативное время ответа;
•	приложение должно давать возможность преподавателю просматривать результаты тестирования учащихся;
•	приложение должно давать возможность учащемуся выбирать тест для прохождения;
•	приложение должно давать возможность учащемуся проходить тесты, отображая на экране вопросы теста;
•	приложение должно давать возможность учащемуся давать ответы на вопросы теста;
•	приложение должно давать возможность учащемуся просматривать результаты своих тестирований;
•	приложение должно давать возможность учащемуся получать рекомендацию учебных материалов согласно результатам теста;
•	приложение должно давать возможность учащемуся просматривать рекомендованные материалы.
К разрабатываемому приложению предъявляются следующие нефункциональные требования:
1.	Требования к функциональной совместимости: веб-приложение должно корректно функционировать в современных веб-браузерах и быть совместимым с основными операционными системами, используемыми на пользовательских устройствах. Интерфейс системы должен поддерживать корректное отображение на экранах различного размера.
2.	Удобство использования: интерфейс веб-приложения должен быть приятен глазу пользователя и интуитивно понятен, между элементами интерфейса должно быть достаточное для комфортного пользования расстояние.
Система интеллектуального подбора учебных материалов будет реализовываться как клиент-серверное веб-приложение. Данное решение было принято для того, чтобы пользователи могли пользоваться системой как с мобильных устройств, так и с помощью компьютера посредством браузера, что потенциально может сильно увеличить как общее количество пользователей системы, так и частоту их пользования.
В общем виде архитектура системы будет состоять из следующих компонентов:
•	клиентская часть – веб-приложение, предоставляющее пользовательский интерфейс (UI) для пользователей системы;
•	серверная часть – приложение, реализующее прикладную логику системы, алгоритм интеллектуального подбора учебных материалов и управление доступом к данным;
•	система хранения данных – база данных, предназначенная для хранения информации о пользователях, учебных материалах, тестах, результатах прохождения тестирования и иных данных системы.
Взаимодействие между клиентской и серверной частями осуществляется по протоколу HTTP с использованием архитектурного стиля REST API. Обмен данными будет производиться в JSON-формате, что будет обеспечивать независимость клиентской и серверной частей и упростит их дальнейшее развитие.
Клиентская часть система отвечает за отображение UI-элементов и их функционирование, ввод и отображение данных, а также за отправку запросов на серверную часть и обработку полученных от сервера ответов. Клиентская часть не содержит бизнес-логику и не выполняет алгоритмических вычислений, связанных с алгоритмом подбора материалов.
Серверная часть системы отвечает за обработку запросов от клиентского приложения, реализацию прикладной логики и взаимодействие с системой хранения данных (база данных). Серверная часть реализуется с использованием многоуровневой архитектуры и предполагает разделение на 3 логических слоя – уровни представления (presentation), прикладной логики (domain) и доступа к данным (data).
Уровень представления реализует программные интерфейсы, через которые клиентская часть взаимодействует с системой. Данный уровень принимает входные запросы, выполняет их первичную валидацию и передаёт управление на уровень прикладной логики.
Уровень прикладной логики является ключевым компонентом всей системы. Здесь реализуются следующие функции:
•	алгоритм подбора учебных материалов;
•	обработка результатов тестирования;
•	формирование рекомендаций для обучающихся;
•	ролевая модель системы.
Уровень доступа к данным отвечает за взаимодействие серверного приложения с базой данных. Он инкапсулирует операции чтения и записи данных, обеспечивая независимость прикладной логики от конкретной реализации базы данных. Таким образом достигается главное преимущество подхода разделения серверной части на 3 логических слоя – все слои работают независимо друг от друга и изменение одного слоя не нарушит логику работы другого слоя.
Система хранения данных предназначена для долговременного хранения информации, используемой в процессе работы системы. В базе данных хранятся данные о пользователях, о тестах, вопросах тестов, результатах тестирований и об учебных материалах. Выделение базы данных в отдельный компонент системы позволяет обеспечить целостность данных, повысить надёжность хранения.
Ниже разберём диаграмму прецедентов (в проекте она изображена в файле .cursor/UseCase.png):
В системе выделяются два основных пользователя: обучающийся и преподаватель. Обучающийся является целевым пользователем системы и использует её для прохождения тестирования, просмотра результатов и получения персонализированных рекомендаций учебных материалов. Преподаватель выполняет функции наполнения и сопровождения системы, связанные с созданием и управлением тестовыми сценариями и образовательным контентом.
Для обучающегося предусмотрены функции, связанные с аутентификацией в системе, прохождением тестирования и анализом полученных результатов. После входа в систему обучающийся может инициировать процесс тестирования, просматривать результаты выполненных тестов, а также получать рекомендации учебных материалов. Процесс получения рекомендаций включает анализ результатов тестирования и определение слабых мест в знаниях с использованием нового алгоритма, после чего осуществляется подбор материалов для дальнейшего изучения. Также обучающемуся доступны функции просмотра учебных материалов и истории прохождения тестов, что позволяет отслеживать динамику обучения и возвращаться к ранее изученному контенту.
Преподаватель взаимодействует с системой посредством функций, связанных с управлением содержимым тестов и рекомендаций. Ему доступны функции добавления учебных материалов, тем, тестов и вопросов теста. Добавление тестов включает создание набора вопросов.
Представленная диаграмма прецедентов демонстрирует функциональную полноту разрабатываемой системы, а также наглядно показывает, какие функции доступны каждой из ролей пользователей – обучающимся и преподавателям.
На рисунке в файле .cursor/Sequence.png представлена диаграмма последовательности, иллюстрирующая процесс взаимодействия пользователя с информационной системой в ходе прохождения тестирования и последующего получения рекомендаций учебных материалов. Диаграмма отражает полный сценарий работы системы – от начала тестирования до отображения результатов и рекомендованного контента. Ниже разберём её подробнее.
Взаимодействие начинается с действий пользователя, который через окно выбора теста инициирует процесс тестирования. Соответствующий запрос передаётся контроллеру тестирования, который выполняет координацию дальнейших шагов и обращается к сервису сессии теста для создания новой тестовой сессии. На этом этапе в системе формируется сущность сессии теста, фиксирующая начало попытки прохождения тестирования.
После создания сессии контроллер тестирования инициирует получение очередного вопроса, обращаясь к сервису вопросов. Сервис осуществляет чтение данных о вопросе из соответствующего хранилища и возвращает эти данные контроллеру, после чего вопрос отображается пользователю в окне вопроса. Данный цикл повторяется для каждого вопроса теста.
При вводе пользователем ответа данные передаются через контроллер тестирования в сервис результатов ответа, который фиксирует информацию о результате. Фиксация результата включает сам введённый ответ, его корректность и временные характеристики ответа. На данном этапе формируется основа для последующего анализа уровня знаний обучающегося.
После завершения тестирования контроллер тестирования обновляет сведения о сессии теста, фиксируя факт её окончания, и инициирует процесс формирования рекомендаций. Для этого осуществляется обращение к модулю рекомендаций, который анализирует накопленные результаты ответов в рамках текущей сессии. На основе этого анализа рекомендательный модуль определяет приоритетные темы и запрашивает соответствующие учебные материалы у соответствующего сервиса.
Сервис учебных материалов выполняет поиск и отбор релевантных материалов, после чего передаёт их рекомендательному модулю. Сформированный список рекомендаций возвращается контроллеру тестирования и отображается пользователю в окне результата. В конце процесса пользователь получает доступ к итогам тестирования и перечню рекомендованных учебных материалов.
Представленная диаграмма последовательности демонстрирует разделение ответственности между пользовательским интерфейсом (UI-слой), бизнес-логикой (domain-слой) и сервисами обработки данных (data-слой), а также подчёркивает роль рекомендательного модуля как ключевого компонента системы. Диаграмма наглядно показывает, каким образом результаты тестирования используются для подбора учебных материалов.
В файле .cursor/Classes.png отражена диаграмма классов. Ниже представлено описание классов, их атрибуты и операции.
Класс User описывает пользователя системы. В системе есть 2 вида пользователей – обучающиеся и преподаватели:
•	id – уникальный идентификатор пользователя;
•	name – имя;
•	login – логин пользователя;
•	password – пароль пользователя;
•	role – роль пользователя в системе (обучающийся или преподаватель).
У данного класса есть и операции:
•	login() – аутентификация пользователя;
•	register() – регистрация пользователя;
•	passTest() – прохождение тестирования;
•	viewResult() – просмотр результатов тестирования;
•	viewRecommendations() – просмотр персональных рекомендаций.
Класс User связан с классом TestSession отношением один ко многим, так как один пользователь может проходить несколько тестовых сессий.
Класс TestSession описывает процесс прохождения теста пользователем по конкретной теме:
•	id – уникальный идентификатор тестовой сессии;
•	userId – идентификатор пользователя;
•	topicId – идентификатор темы тестирования;
•	startedAt – дата и время начала тестирования;
•	finishedAt – дата и время конца тестирования;
•	score – итоговый балл за тест.
Каждая тестовая сессия связана с одним пользователем и с несколькими объектами класса AnswerResult, отражающими ответы на отдельные вопросы.
Класс AnswerResult предназначен для хранения информации об ответе обучающегося на конкретный вопрос в рамках тестовой сессии:
•	id – уникальный идентификатор результата ответа;
•	testSessionId – идентификатор тестовой сессии;
•	questionId – идентификатор вопроса;
•	userAnswer – пользовательский ответ;
•	isCorrect – признак правильности ответа;
•	score – балл, полученный за данный ответ;
•	responseTimeMs – время ответа в миллисекундах.
Данный класс позволяет учитывать не только корректность ответа, но и дополнительные характеристики, такие как время, которое используется в алгоритме анализа знаний.
Класс Question описывает вопрос теста:
•	id – уникальный идентификатор вопроса;
•	topicId – идентификатор темы;
•	text – текст вопроса;
•	rightAnswer – эталонный ответ (для вариантов с выбором — перечень верных вариантов или ключ; для текстового ответа — возможна заготовка или пустое значение, если корректность определяется иначе);
•	difficulty – показатель сложности вопроса;
•	answerType – формат ответа учащегося: один вариант из списка; несколько вариантов; свободный текст (см. раздел «Проектные решения»).
Каждый вопрос относится к одной теме.
Класс Topic представляет учебную тему, в рамках которой группируются вопросы и учебные материалы:
•	id – уникальный идентификатор темы;
•	name – название темы;
•	description – описание темы.
С каждым объектом класса Topic связаны:
•	несколько вопросов;
•	несколько учебных материалов.
Отдельная сущность «Тест» в модели не вводится: единицей проверки знаний считается тема (`Topic`). Набор вопросов по теме вместе с прохождением в `TestSession` с полем `topicId` соответствует понятию «тест по теме» в требованиях. При необходимости в будущем можно добавить сущность Test как группу вопросов или несколько тестовых наборов на одну тему без ломки обучающей части алгоритма.
Класс Material описывает учебные материалы, которые рекомендуются для изучения темы:
•	id – уникальный идентификатор материала;
•	topicId – идентификатор темы;
•	title – название материала;
•	link – URL или внутренний путь к контенту;
•	type – вид представления материала: текстовая статья (в т.ч. HTML/Markdown на странице), видео; при необходимости расширяется значениями вроде внешней ссылки или файла (см. раздел «Проектные решения»).
Учебные материалы используются при формировании персональных рекомендаций для обучающихся по результатам тестирования.
Ниже рассмотрим непосредственно алгоритм рекомендаций.
В условиях практического применения образовательных информационных систем актуальной является задача разработки алгоритма, позволяющего учитывать совокупность параметров ответов обучающегося и при этом сохраняющего интерпретируемость и простоту реализации. В связи с этим в рамках работы предлагается описать алгоритм подбора учебных материалов, основанный на комплексной оценке уровня усвоения учебного материала по отдельным темам.
Новый алгоритм интеллектуального подбора учебных материалов основан на оценке уровня усвоения темы, которая вычисляется на основе совокупности показателей ответов обучающегося. Используемые показатели:
	правильность ответа;
	время реакции;
	относительная сложность вопроса.
Алгоритм формирует динамический профиль обучающегося по темам, приоритизирует темы с наибольшим образовательным дефицитом и на этой основе рекомендует учебные материалы подходящего уровня сложности.
В отличие от ранее рассмотренных классических моделей, предлагаемый алгоритм:
	не использует вероятностные скрытые модели;
	не требует обучения на больших выборках;
	ориентирован на прямую интеграцию в прикладную информационную систему.
Далее будут рассмотрены конкретные формулы, используемые в данном алгоритме.
Для каждого ответа обучающегося фиксируются следующие параметры:
	ai – индикатор правильности ответа;
	ti – время ответа на вопрос;
	di – коэффициент сложности вопроса;
	Ti – нормативное время ответа на вопрос.
Значения ai рассчитываются следующим образом:
a_i  = {(1,если ответ верный, и 0,если ответ неверный)}
Значения di рассчитываются следующим образом:
d_i=1-C_i/N_i 
В данной формуле Ci – количество правильных ответов на вопрос i, Ni – общее количество попыток ответа на вопрос i. Таким образом, чем ближе значение di к значению 1, тем сложнее считается вопрос, чем ближе к 0 – тем он проще.
Значения Ti рассчитываются следующим образом:
T_i=median(t_(i,1),t_(i,2),…,t_(i,N_i ) )
В данной формуле ti,k – время ответа k-го пользователя на вопрос i. Таким образом медианное время ответа пользователей на вопрос будет считаться нормативным.
Вводится нормированная временная характеристика:
v_i  =   min⁡(1,T_i/t_i )
По этой формуле быстрый и уверенный ответ даёт значение, близкое к 1, в то время как медленный ответ снижает итоговую оценку, а значение всегда лежит в диапазоне от близкого к 0 до 1.
Для каждого ответа вычисляется частичная оценка по формуле:
q_i  =(a_i+ v_i+ d_i)/3
Таким образом учитываются сразу несколько критериев:
	скорость ответа;
	сложность вопроса;
	факт знания.
Для курса C, содержащего n вопросов, вычисляется уровень усвоения:
M_C  =  1/n ∑_(q_i) 
Значение MC в данном случае показывает оценку усвоения по темам (курсам).
Также вводятся два пороговых значения:
	hlow – нижний порог усвоения темы;
	hhigh – верхний порог усвоения темы.
Далее сравниваются значения MC с пороговыми значениями:
	MC < hlow – тема требует повторного изучения;
	hlow ≤ MC < hhigh – тема требует закрепления;
	MC ≥ hhigh – тема усвоена.
После этого рекомендуются материалы, уровень которых является подходящим для определённой степени усвоения темы.

## Проектные решения (согласовано)

**Тема как «тест».** Отдельная сущность Test не описывается: проверка привязана к `Topic`, сессия — к `TestSession.topicId`, вопросы — к `Question.topicId`. В интерфейсе формулировки «выбрать тест» и «тест по теме» эквивалентны выбору темы. Если позже понадобится несколько независимых наборов вопросов на одну тему, можно ввести сущность Test или QuestionSet без изменения смысла алгоритма по `topicId`.

**Формат ответа (`Question.answerType`).** Унифицированные значения (перечисление на API и в БД):

| Значение | Смысл |
|----------|--------|
| `single` | Выбор одного варианта из списка |
| `multiple` | Выбор нескольких вариантов |
| `text` | Ввод текста (короткий или развёрнутый ответ) |

В коде и API полезно называть поле **`answerType`**, а не «тип вопроса», чтобы не смешивать с педагогическим типом задания. Для `text` нужно заранее определить на сервере правило выставления `isCorrect` и `ai` (точное совпадение, регистронезависимое сравнение, ручная проверка преподавателем позже и т.д.) — иначе автоматическая оценка и алгоритм по `ai` теряют смысл.

**Материалы (`Material.type`).** Базовый набор: **`article`** (текст/статья на сайте, HTML или Markdown) и **`video`** (встроенный плеер или ссылка на видеохостинг). Расширение без переделки модели: **`external`** — произвольная внешняя ссылка (PDF, курс), **`document`** — загруженный файл. Для НИР достаточно `article` + `video`; остальное — по мере необходимости.

**Аутентификация.** Используется **JWT**: клиент передаёт токен в заголовке `Authorization: Bearer <accessToken>`. Срок жизни access-токена и при необходимости refresh-токен и хранение (память, `localStorage`, httpOnly-cookie) согласуются при реализации backend; клиент при 401 очищает сессию и перенаправляет на вход.

**REST API.** Полного перечня эндпоинтов нет; вводятся по мере разработки, начиная с основных (логин, темы, сессия теста, ответы, рекомендации). Стиль — REST, JSON.

## Замечания к проектированию (рекомендации)

1. **Пароль в модели `User`:** в БД хранить только хэш (bcrypt/argon2), не открытый пароль; в API не возвращать поле password.
2. **Варианты ответов:** для `single`/`multiple` хранить варианты отдельно (таблица или JSON), а `rightAnswer` трактовать как id/набор id верных вариантов — так проще редактировать и валидировать.
3. **Нормативное время `Ti`:** для новых вопросов с малым `Ni` медиана нестабильна; на старте можно задавать `Ti` вручную преподавателем или брать дефолт до накопления статистики.
4. **Связь рекомендаций с «уровнем материала»:** в тексте алгоритма фигурируют пороги усвоения; для `Material` стоит явно завести поле уровня/сложности или тег, иначе подбор «материала под степень усвоения» остаётся на усмотрение кода без опоры в данных.
5. **Диаграммы:** прецеденты, последовательность и классы — в файлах `.cursor/UseCase.png`, `.cursor/Sequence.png`, `.cursor/Classes.png`.

## Frontend: назначение и границы

Клиент отвечает только за UI, ввод данных, отображение состояний и вызовы API; расчёт рекомендаций и порогов `hlow` / `hhigh` выполняются на сервере.

## Frontend: стек (зафиксировать при старте репозитория)

| Решение | Комментарий |
|--------|-------------|
| Сборка / шаблон | Например Vite + выбранный UI-фреймворк (React / Vue / Svelte — указать выбранный). |
| Маршрутизация | Отдельные зоны маршрутов для роли «обучающийся» и «преподаватель» (и общие: вход, регистрация). |
| HTTP-клиент | `fetch` или обёртка (axios и т.д.); единая база URL API и обработка ошибок. |
| Аутентификация | JWT: заголовок `Authorization: Bearer …`; хранение access-токена (например `localStorage` или память) и обработка 401 — согласовать с backend; при наличии refresh — обновление пары токенов. |
| Состояние | Локально по страницам и/или общий store — по сложности экранов. |

## Frontend: ключевые экраны и потоки

**Обучающийся**

- Вход / регистрация (если предусмотрена).
- Список доступных тестов (выбор теста → старт сессии).
- Прохождение теста: пошаговый вопрос, ввод ответа, при необходимости таймер или отображение нормативного времени (если сервер отдаёт `Ti` или аналог).
- Экран итога сессии: балл, краткая сводка, блок рекомендованных материалов.
- История своих тестов и просмотр детализации результатов (в рамках требований).
- Просмотр рекомендованных и прочих учебных материалов (переход по `link`, тип `Material.type`).

**Преподаватель**

- Управление темами (`Topic`): список, создание, редактирование.
- Управление материалами (`Material`) в привязке к теме.
- Управление вопросами по теме: редактор с полями сложности, нормативного времени, формата ответа (`answerType`: один / несколько / текст) и вариантами для выбора; привязка к теме (тест = тема).
- Просмотр результатов тестирования учащихся (агрегаты и при необходимости детализация по `AnswerResult`).

**Общее**

- Адаптивная вёрстка (из нефункциональных требований): минимум одна мобильная и одна десктопная ширина в макетах.
- Явные состояния загрузки и ошибок API (сеть, 401, валидация).
